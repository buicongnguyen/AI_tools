Prompt engineering

A good prompt is an executable engineering brief

For important tasks, use six blocks. Add only the detail that changes the result.

1. GoalObservable outcome
2. ContextSystem, files, symptom
3. ConstraintsScope, safety, compatibility
4. ReferencesPatterns, logs, designs, specs
5. VerificationTests, metrics, measurements
6. OutputEvidence and remaining risks
Weak

“Add notifications and make the UI modern.”

No user outcome, boundaries, reference, or way to prove completion.

Stronger

“Notify a signed-in user when their export finishes. Follow the billing-alert pattern, preserve the existing queue, test authorization and duplicate events, run the browser flow, and report migration and rollback evidence.”

Coding: implement a software feature 
Goal:
Add an in-app notification when a background export finishes. A signed-in user
must see only their own completed exports and can mark a notification as read.

Context:
Read AGENTS.md and trace the existing export job, authorization, event handling,
and the billing-alert notification pattern.

Constraints:
Preserve the public API and existing queue. Avoid broad refactoring or new
dependencies. Never expose another user's export metadata.

References:
Follow the billing-alert implementation, migration style, and design-system
component patterns already in the repository.

Verification:
Test success, duplicate events, authorization, empty state, unread count, and
mark-as-read. Run focused tests, type check, lint, and the critical browser flow.

Output:
Report changed files, data flow, commands/results, migration and rollback,
security review, assumptions, and remaining risks.
Software engineer: diagnose a production bug 
Goal:
Find and fix the root cause of login failure after an access token expires.

Context:
Users remain on the spinner after about 60 minutes. A network trace shows a 401
from /session/refresh followed by repeated profile requests. Confirm the real flow.

Constraints:
Diagnose before editing. Do not hide the 401, extend token lifetime, add blind
retries, or weaken authentication.

References:
Use the captured network trace, auth integration tests, and git history for the
refresh-token implementation.

Verification:
Reproduce the failure, rank hypotheses, add a failing regression test, implement
the smallest root-cause fix, then run auth tests and the browser expiry flow.

Output:
Report reproduction, root cause, before/after evidence, changed files, commands,
security impact, and unresolved edge cases.
Software architecture: make a design decision 
Goal:
Recommend the simplest architecture for [CAPABILITY] that meets the next
12 months of requirements.

Context:
Current architecture, expected load, SLO, team skills, integrations, and known
growth are [DETAILS].

Constraints:
Preserve [COMPATIBILITY]. Include migration, failure handling, operations,
observability, cost, and rollback—not only a component diagram.

References:
Inspect the existing repository and deployment. Verify current service limits
and APIs using primary official documentation.

Verification:
Compare at least three options across latency, throughput, consistency,
availability, security, complexity, cost, migration, and failure scenarios.

Output:
Write an ADR with an option table, recommendation, diagrams, rejected
alternatives, migration phases, test plan, risks, and revisit conditions.
Design system: audit and improve consistency 
Goal:
Audit the current design system and propose a bounded plan to improve
consistency, accessibility, and developer usability.

Context:
Inspect tokens, components, Storybook/examples, CSS, approved designs, and the
three most-used product flows.

Constraints:
Do not redesign the brand or replace the framework. Preserve public component
APIs unless migration is justified. Meet WCAG 2.2 AA and support keyboard,
dark mode, responsive layouts, reduced motion, and localization.

References:
Compare implementation to approved designs and existing product patterns, not
personal taste.

Verification:
Inventory tokens/components, find duplicate values and missing states, sample
real screens at mobile/desktop widths, run accessibility checks, and manually
test focus and keyboard behavior.

Output:
Return a ranked audit, proposed token hierarchy, component-state matrix,
before/after examples, migration phases, risks, and measurable completion criteria.

AI harness

The model thinks; the harness makes engineering work possible

An AI harness is the controlled system around a model: context, tools, execution, permissions, verification, memory, and feedback. Claude Code is the harness around Claude; Codex provides the same kind of agent runtime around OpenAI models.

Repository files, concise instructions, specifications, decisions, and relevant history.

Use a harness efficiently

Make the project legible

Document exact build, test, simulation, run, and release commands plus architecture and constraints.

Close the feedback loop

Give the agent evidence it can inspect and objective pass criteria it can iterate against.

Treat context as a budget

Keep persistent instructions short; send noisy research to isolated tasks and request summaries.

Bound autonomy

Allow routine work inside a sandbox; require approval for destructive, external, or production actions.

Separate judgment and enforcement

Use skills for reasoning workflows and hooks/CI for rules that must execute every time.

Make the system observable

Expose structured logs, traces, screenshots, reports, events, and measurements to the agent.

Harness maturity ladder

Level 0Chat only. The human copies context, executes everything, and checks results manually.
Level 1Tool access. The agent reads, edits, and runs commands inside a controlled repository.
Level 2Self-verifying. Automated tests, browser checks, simulations, or measurements feed the loop.
Level 3Repeatable. Skills, hooks, templates, and CI preserve reliable workflows.
Level 4Parallel. Isolated agents and worktrees handle independent tasks with coordinated review.
Level 5Operational. Deployment, monitoring, rollback, audit, and policy are integrated.
Do not jump directly to high autonomy. Increase autonomy only after the current level repeatedly produces correct, reviewable evidence inside safe boundaries.

Realistic cases

Choose the engineering situation closest to yours

Ship a device-health dashboard without inventing the architecture

Situation: technicians need to see offline devices, error trends, and firmware versions. The repository already has telemetry ingestion, but nobody knows the full data path.

Agent job: trace ingestion to storage and UI, identify existing patterns, propose the smallest read-only dashboard slice, implement it, add contract and browser tests, and prepare a preview.

Useful result: one real fleet query, one tested API route, one accessible screen, a measured response-time budget, and a preview URL—not a speculative platform rewrite.

Evidence to require

  • Data-flow map with file references
  • API contract and permission test
  • Empty/error/loading states
  • Browser screenshot and accessibility check
  • Preview smoke test and rollback

Debug an intermittent I²C sensor lockup

Situation: a sensor node freezes after hours in a thermal chamber. Application logs show timeouts, but the root cause could be task scheduling, bus recovery, power integrity, or the device itself.

Agent job: correlate host timestamps, driver logs, firmware event markers, logic-analyzer traces, rail measurements, board revision, and temperature; rank hypotheses and design one distinguishing experiment per cause.

Useful result: the first layer where behavior diverges, a reproducible fault-injection test, a bounded recovery change, and repeated chamber evidence.

Evidence to require

  • Exact board and firmware revision
  • SDA/SCL and rail capture
  • Timeout/retry state timeline
  • Bus-stuck recovery test
  • Temperature and duration matrix

Improve accelerator throughput without breaking correctness

Situation: an FPGA inference block meets functional simulation but misses throughput and occasionally corrupts output under DMA backpressure.

Agent job: read interface specifications and timing reports, produce a latency/throughput model, add protocol assertions and randomized backpressure, locate the first invalid handshake, and compare two pipeline fixes.

Useful result: a self-checking failing test, waveform-linked root cause, post-route timing evidence, resource delta, and measured board throughput.

Evidence to require

  • Golden reference vectors
  • Ready/valid assertions
  • Random stalls and reset tests
  • Timing and CDC reports
  • Post-route and board measurements

Efficiency system

Spend tokens on evidence, not repeated correction

1. Define “done”

Give the agent observable acceptance criteria, named files, boundaries, and exact verification commands.

2. Protect context

Keep instruction files concise, use focused tests, summarize noisy exploration, and start fresh when history stops helping.

3. Close the loop

Let the agent run tests, inspect screenshots, compare outputs, and fix discrepancies before asking you to review.

4. Systemize wins

Stable rules go in AGENTS.md or CLAUDE.md; repeatable workflows become skills; mandatory checks become hooks or CI.

Software track

Idea to production

  1. Turn the user problem into acceptance tests.
  2. Trace architecture, interfaces, data, and trust boundaries.
  3. Build the smallest vertical slice.
  4. Run unit, integration, end-to-end, performance, and security checks.
  5. Debug from reproduction evidence, not guesses.
  6. Review, stage, deploy, observe, and preserve rollback.
Hardware track

Idea to fielded system

  1. Define performance, power, thermal, area, cost, and safety budgets.
  2. Document clocks, resets, protocols, register maps, and failure behavior.
  3. Model or simulate before committing to hardware.
  4. Build RTL, firmware, PCB, or prototype with self-checking tests.
  5. Bring up safely and correlate logs, traces, waveforms, and measurements.
  6. Stress, fault-test, pilot, update, monitor, and verify recovery.
Reality checks

Where AI engineering work usually fails

Wrong problem

The agent builds what was requested, but the request never represented a real user or system need.

Countermeasure: require problem evidence and a falsifiable prototype.

Context overload

Long logs, broad scans, and stale instructions crowd out the decisions that matter.

Countermeasure: focused commands, summaries, concise instruction files, fresh review contexts.

False completion

“Build passed” is reported even though the real workflow, failure modes, timing, or hardware were never exercised.

Countermeasure: define evidence and pass criteria before implementation.

Automation too early

An unstable manual process is scaled into noisy CI or unattended agents.

Countermeasure: prove the workflow on small samples, then turn it into a skill, hook, or automation.

Builder bias

The same context that created the solution rationalizes its mistakes.

Countermeasure: use a fresh reviewer with requirements, diff, and evidence only.

Hardware optimism

Simulation or datasheet reading is mistaken for electrical, timing, thermal, or field validation.

Countermeasure: calibrated bench evidence, tool reports, HIL, stress, and recovery testing.
