Case 1 · Debugging

Users are randomly logged out during token refresh

Intermediate60–90 min
Ticket: A React + Node application sends several refresh requests when API calls race near token expiry. One rotates the refresh token; a late call fails and logs the user out. Slow networks expose it, but current unit tests do not.
Browser
3 calls see expiry
Interceptor
starts 3 refreshes
Auth service
rotates once
Failure
late call logs out

Context

  • src/api/client.ts
  • src/auth/session.ts
  • Auth tests and rotation contract

Acceptance

  • One refresh per session.
  • Waiters resume with new token.
  • Failure logs out once.
  • Deterministic regression.
  • No token values in logs.

Wrong fixes

  • Disable rotation.
  • Add sleeps or broad retries.
  • Swallow 401s.
  • Use one lock for all users.

Claude Code path

  1. Read CLAUDE.md, auth flow, tests, and contract.
  2. Use Plan mode to draw concurrent state before editing.
  3. Create a deterministic failure with deferred promises/barriers.
  4. Implement a session-scoped single-flight promise.
  5. Use a fresh subagent for race and secret review.

Codex path

  1. Read applicable AGENTS.md and trace auth flow.
  2. Use /plan for invariants and reproduction.
  3. Require the baseline-failing concurrency test.
  4. Make the smallest change and run client tests.
  5. Run /review against ticket, diff, and output.
Copy-ready prompt
Goal: Find and fix the refresh-token race that randomly logs users out.

Inspect repository instructions, client/session code, auth tests, and the
rotation contract. Before editing, draw the concurrent sequence, state the
session invariants, and add a deterministic regression using controlled
promises/barriers. It must fail on current code without sleeps.

Constraints: preserve rotation and public APIs; never log tokens; do not swallow
401s or add broad retries. Done: one refresh per session, waiters resume, failure
logs out once, focused and broader tests pass, and fresh review finds no
unresolved concurrency or secret issue. Return evidence and residual risks.
Case 2 · Backend

Add idempotent payment-webhook processing

Intermediate90–120 min
Ticket: A payment provider retries events for 72 hours. The endpoint updates an order and sends an email on every delivery. Duplicate and out-of-order events have caused repeated emails and an invalid status regression.
Provider
signed event ID
HTTP edge
verify raw body
Transaction
claim + transition

Acceptance

  • Verify raw-body signature.
  • Event ID changes state once.
  • Older event cannot regress state.
  • Notification is created once.
  • Concurrent duplicate test.

Decisions

  • Unique provider/event constraint
  • State-transition table
  • Transaction boundary
  • Outbox and retention

Wrong fixes

  • In-memory seen-ID set.
  • Trust timestamp ordering.
  • Return 200 before commit.
  • Email inside unsafe retry.

Claude Code path

  1. Plan route, ORM transactions, schema, and outbox patterns.
  2. Define invariant and transaction diagram.
  3. Implement migration, handler, and tests as a vertical slice.
  4. Use a security subagent for signature/replay review.

Codex path

  1. Use /plan to trace body handling and state rules.
  2. Write duplicate, concurrent, out-of-order, and invalid-signature tests.
  3. Implement the migration and slice in a worktree if needed.
  4. Use /review for transaction safety and rollback.
Copy-ready prompt
Goal: Make payment webhooks safe under duplicate, concurrent, delayed, and
out-of-order delivery while preserving order APIs.

Inspect the route, raw-body/signature middleware, state machine, transaction
patterns, outbox/email code, migrations, and tests. Plan event identity, durable
uniqueness, allowed transitions, transaction boundary, side effects, retention,
and rollback before editing.

Acceptance: invalid signatures make no change; concurrent duplicate applies
once; older events cannot regress state; notification is produced once after
commit. Implement the smallest vertical slice, add integration tests, run schema
and backend checks, then perform fresh security/correctness review.
Case 3 · Performance

Reduce dashboard API p95 without hiding the bottleneck

Advanced90–150 min
Ticket: GET /api/projects/:id/dashboard has p95 780 ms at 30 RPS and makes 101 queries for 50 widgets. Target: p95 ≤ 250 ms without stale authorization or an unbounded cache.

Baseline

  • Fixed dataset/seed
  • Query count
  • p50/p95/p99
  • CPU, memory, DB time
  • Authorized/forbidden cases

Acceptance

  • ≤5 queries.
  • p95 ≤250 ms at 30 RPS.
  • Authorization unchanged.
  • Response contract unchanged.
  • Benchmark committed.

Wrong fixes

  • Cache everything forever.
  • Measure one warm request.
  • Remove auth joins.
  • Change data between runs.

Claude Code path

  1. Inspect traces, ORM, query logs, and benchmark tools.
  2. Use Plan mode to separate query, serialization, and lock hypotheses.
  3. Create fixed-data benchmark and query assertion.
  4. Change one query shape and compare baseline/candidate.

Codex path

  1. Use /plan to map fan-out, auth, and measurement.
  2. Create a reproducible benchmark artifact and baseline report.
  3. Implement one optimization and retain before/after data.
  4. Use /review for auth, memory, contract, and validity.
Copy-ready prompt
Goal: Reduce dashboard p95 to ≤250 ms at 30 RPS while preserving authorization
and the response contract.

Do not optimize yet. Inspect tracing, ORM/query code, serialization, indexes,
auth, dataset shape, and benchmark tooling. Build a fixed baseline recording
query count, p50/p95/p99, CPU, memory, and DB time for allowed/forbidden users.
Rank bottlenecks and run the cheapest distinguishing measurement.

Implement one bounded change. Do not add an unbounded cache or remove security
checks. Done: ≤5 queries, p95 target met, unchanged contract/auth tests, committed
benchmark, and fresh review of correctness and measurement validity.
Case 4 · Frontend

Build optimistic settings that safely roll back

Intermediate60–100 min
Ticket: Notification settings should feel instant, but the API can reject policy changes or fail offline. Rapid toggles let late responses overwrite newer intent, and screen-reader users do not hear save failures.

States

  • Saved server snapshot
  • Optimistic value
  • Mutation identity
  • Pending/offline
  • Error/retry

Acceptance

  • Latest intent wins.
  • Rejection restores server truth.
  • Offline retry is visible.
  • Error is announced.
  • Focus stays predictable.

Wrong fixes

  • Disable all controls.
  • Rollback to stale value.
  • Ignore late responses.
  • Color-only error toast.

Claude Code path

  1. Inspect state, API, design system, and accessibility tests.
  2. Use Plan mode for the overlapping-request state machine.
  3. Implement one preference with controlled responses.
  4. Run browser checks and fresh accessibility review.

Codex path

  1. Trace component, request/cache layer, and a11y patterns.
  2. Use /plan for optimistic/server state and ordering.
  3. Implement the slice and run component/browser tests.
  4. Use /review for race, focus, announcements, recovery.
Copy-ready prompt
Goal: Implement optimistic notification settings where newest intent wins and
failed changes recover accessibly.

Inspect the component, API/cache layer, design-system patterns, announcements,
and tests. Before editing, model saved, optimistic, pending, error, offline, and
overlapping request states. Draw two rapid toggles with reversed responses.

Acceptance: instant update; late response cannot overwrite newer intent; policy
rejection restores server truth; offline state is retryable; error has an
accessible live announcement; focus is stable. Add deterministic success,
rejection, offline, and reversed-response tests plus browser/a11y evidence.

Copy-ready prompt
Goal: Migrate users.full_name to display_name with zero downtime, mixed-version
compatibility, and a verified 24-hour rollback window.

Inspect schema history, every reader/writer, serializers, jobs/reports, deployment
order, data size, and rollback. Do not edit before a compatibility matrix and
expand/contract plan. Plan additive schema, compatible reads/writes, bounded
resumable backfill, parity metrics, consumer migration, rollback rehearsal, and
a separately approved drop release.

Implement only the approved phase. Add mixed-version contracts, migration tests,
observability, and operator runbook. Report locks, duration, rollback commands,
raw results, and remaining consumers.
Case 6 · Incident

Stop consumer memory growth without losing messages

Advanced90–180 min
Incident: A Node event consumer grows from 300 MB to 2.2 GB in four hours, then restarts. Queue lag spikes. Heap snapshots retain messages, but the cause may be unbounded concurrency, listeners, failed acknowledgements, or tracing baggage.

Evidence

  • Heap snapshots over time
  • Retainer paths
  • In-flight and lag
  • Ack/nack/retry metrics
  • Listeners and timers

Acceptance

  • Bounded steady memory.
  • No lost/duplicate effect.
  • Bounded in-flight work.
  • Safe drain/requeue.
  • Canary rollback thresholds.

Wrong fixes

  • Only increase memory.
  • Force periodic GC.
  • Ack before completion.
  • Reduce concurrency blindly.

Claude Code path

  1. Analyze heap, queue metrics, lifecycle, and ack semantics.
  2. Use Plan mode for ranked hypotheses and load experiments.
  3. Add soak test and observability before the fix.
  4. Stop at canary gate after fresh reliability review.

Codex path

  1. Inspect lifecycle, queue client, tracing, shutdown, artifacts.
  2. Use /plan for hypotheses, soak, and rollback thresholds.
  3. Implement backpressure/lifecycle correction with evidence.
  4. Use /review; stop for human canary approval.
Copy-ready prompt
Goal: Fix event-consumer memory growth without message loss, duplicate side
effects, or unacceptable throughput regression.

Inspect heap/retainer paths, in-flight/lag metrics, ack/nack semantics, listeners,
timers, tracing, shutdown, and recent diffs. Do not propose a fix until evidence
separates concurrency, listener, retained-failure, and tracing hypotheses.

Build a fixed-workload soak test recording memory, GC, in-flight, throughput, lag,
errors, and delivery. Done: bounded memory, preserved delivery, backpressure,
safe shutdown, acceptable throughput, and canary rollback thresholds. Stop before
production deployment and request human approval.
