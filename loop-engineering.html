What it means

A prompt asks once. A loop keeps returning to the work.

Loop Engineering is the practice of designing repeatable, bounded feedback cycles around AI agents. The loop finds or receives work, gives an agent the right state and tools, checks the result with real evidence, saves what happened, and decides whether to iterate, stop, or request human judgment.

Important: “Loop Engineering” is an emerging mid-2026 practitioner term, not a finalized industry standard. The underlying ideas—agent loops, feedback, evaluators, state, budgets, isolation, and human gates—are established and appear in current OpenAI and Anthropic engineering guidance.

Prompt engineering

Optimizes one instruction.

Output: a better request.

Context engineering

Optimizes what one model run can see.

Output: relevant files, facts, and constraints.

Harness engineering

Optimizes one agent run.

Output: tools, permissions, execution, and verification.

Loop engineering

Optimizes repeated agent work.

Output: triggers, state, isolation, agents, gates, and stop rules.
Deep system model

There are three loops, operating at different speeds

An agent already has an internal reason–act–observe cycle. Loop Engineering adds an external, inspectable control loop around one or more agent runs. A third loop connects shipped results to real users and operating data. Keep their state, evidence, and stop rules separate.

1. Agent action loop

Read context → choose tool → act → observe. The harness manages this inside one session.

Stops: response, tool limit, error, or agent judgment.

2. Engineering loop

Shape task → make change → verify → retry or hand off. Your controller owns this.

Stops: evidence, policy, risk, or budget.

3. Product learning loop

Deploy → observe telemetry/users → evaluate impact → change backlog, tests, or harness.

Stops: product decision or next experiment.

Minimal mathematical model

Sₜ = {goal, attempt, workspace, hypothesis, evidence, budget, policy}
Uₜ = Maker(goal, Sₜ) → candidate change
Eₜ₊₁ = Verify(candidate, immutable requirements, protected checks)
Dₜ₊₁ = Controller(Eₜ₊₁, progress, risk, remaining budget)
Sₜ₊₁ = Record(Sₜ, Uₜ, Eₜ₊₁, Dₜ₊₁)

The model matters because it locates responsibility. The maker proposes an action. The verifier produces evidence. The controller—not the maker—decides whether the run passes, retries, blocks, escalates, or exhausts its budget.

Named state transitions

NEW→ READY→ RUNNING→ VERIFYING→ PASSED
VERIFYING→ RETRYABLE↺ READY · BLOCKED ESCALATEDBUDGET_EXHAUSTED
Design rule: every run must end in exactly one named terminal or handoff state. “Still trying,” “probably fixed,” and “looks good” are not states.

1. Trigger

Schedule, issue, alert, failed check, deployment event, measurement, or human request.

2. State

Goal, current attempt, completed work, evidence, failures, decisions, and remaining budget.

3. Workspace

Disposable branch/worktree or sandbox with scoped network, secrets, and tools.

4. Maker

The agent that investigates and produces the next bounded change or experiment.

5. Verifier

Independent tests, measurements, policies, and preferably a fresh reviewer context.

6. Controller

Applies stop conditions, retry policy, budgets, human gates, and state transitions.

Implementation guide

Build the smallest safe loop first

0

Prove the workflow manually

Run one task with a human steering the agent. Record the prompt, context, commands, evidence, judgment, and recovery. Do not automate a workflow you cannot yet review.

1

Define the goal and terminal states

Write machine-checkable success where possible. Define passed, retryable failure, blocked, budget exhausted, and human approval required.

2

Externalize loop state

Store state outside conversation history: task ID, branch, attempt number, evidence, last failure, decisions, and next action. Use a JSON/Markdown artifact, database record, issue, or workflow state.

3

Isolate every attempt

Use a clean branch, worktree, container, or cloud environment. Pin setup and dependency versions. Never let parallel makers silently edit the same checkout.

4

Write a bounded maker contract

Give the maker one outcome, relevant state, allowed tools, constraints, evidence requirements, and maximum scope. Ask it to return a patch or experiment—not a declaration of success.

5

Build independent verification

Prefer deterministic tests and real measurements. Separate generator and evaluator roles. The verifier receives requirements, the candidate change, and evidence—not the maker’s persuasive narrative.

6

Add budgets and recovery

Limit attempts, wall time, tokens/cost, changed files, commands, and concurrency. Save partial evidence, clean abandoned workspaces, and escalate repeated or unsafe failures.

7

Introduce triggers last

Start on demand. Then add GitHub events, alerts, schedules, or APIs only after the loop reliably stops, reports, and preserves review boundaries.

Reference design

A loop is a state machine, not an infinite prompt

The following loop.yml is an illustrative design schema—not a standard file format for Claude Code or Codex.

name: repair-failing-ci
trigger:
  event: github.check_failed

workspace:
  isolation: disposable-worktree
  network: allowlisted
  secrets: read-only-ci-token

state:
  store: .loop/runs/{run_id}.json
  fields: [task, attempt, branch, evidence, last_failure, decision]

maker:
  goal: "Produce the smallest root-cause fix and regression test"
  constraints:
    max_changed_files: 8
    forbidden: [production_deploy, secret_output, dependency_major_upgrade]

verifiers:
  - command: npm test -- --runInBand
  - command: npm run typecheck
  - command: npm run lint
  - agent_review: correctness-security-scope

budget:
  max_attempts: 4
  max_minutes: 45
  max_cost_usd: 12

decision:
  pass: open_draft_pr
  retryable: feed_failure_to_next_attempt
  repeated_failure: escalate_with_evidence
  risky_change: require_human_approval
  budget_exhausted: stop_and_report

Controller pseudocode

while budget.remaining() and not state.terminal:
    signal = discover_or_resume(state)
    task = shape_task(signal, state, policy)
    candidate = maker.run(task, isolated_workspace())
    evidence = deterministic_checks(candidate)
    review = independent_verifier(candidate, evidence, requirements)
    decision = controller.decide(evidence, review, policy)
    state.record(candidate, evidence, review, decision)

    if decision == PASS:
        open_draft_change_for_human_review()
        break
    if decision in [BLOCKED, UNSAFE, BUDGET_EXHAUSTED]:
        escalate_with_evidence()
        break
    prepare_next_attempt_from_specific_failures()

Convergence engineering

Make every retry earn the right to exist

A bounded loop should move toward a measurable acceptance region. It should not merely generate more activity. Define an acceptance vector rather than one vague score, because an average can hide a critical regression.

Acceptance vector dimensions, thresholds, and protected invariants
Acceptance dimensionExample thresholdInvariant that cannot regress
Functional correctnessTarget regression and affected suite passNo test deletion, skip, or weakened assertion
Scope≤ 8 files; approved paths onlyNo unrelated refactor or generated-file churn
Performancep95 ≤ 250 ms; memory ≤ 300 MBBaseline error rate and throughput retained
SecurityNo new high-severity findingNo permission, secret, or network expansion
Hardware3 repeated HIL trials within toleranceSafety limits, timing margin, and thermal ceiling retained

The retry contract

1. Specific failure

Retry only from a reproducible failure signature, a reviewer finding, or a measured gap—not a generic “improve it.”

2. Changed hypothesis

The next attempt must state what it now believes and how that differs from the previous attempt.

3. Smallest experiment

Prefer a diagnostic or narrow patch that can disprove the hypothesis before a large implementation.

4. Regression comparison

Compare the new evidence with the last best candidate. Roll back changes that move any protected dimension backward.

5. Patience window

Stop after the same normalized failure repeats, after no metric improves for N attempts, or when evidence is too noisy.

6. Capability diagnosis

Escalate whether the missing piece is context, permission, tool, environment, verifier, requirement, or human judgment.

progressₜ = accepted_dimensionsₜ − regressionsₜ
retry only if: failure is reproducible AND hypothesis changed AND budget remains
stop if: pass OR unsafe OR blocked OR repeated_failure ≥ patience OR budget exhausted
For noisy systems: use repeated measurements, confidence bands, warm-up rules, and a required number of consecutive passes. One lucky benchmark or one intermittent hardware run is not convergence.
Verifier architecture

Verification is a stack, not a single critic prompt

No one layer catches every defect. Build from deterministic checks upward, and require the layers appropriate to the change’s risk. AI review is useful for semantic gaps, but it should not replace executable or physical evidence.

Verification levels, evidence, strengths, and weaknesses
LevelEvidenceGood forWeakness
L0 · Maker reportClaim, diff summary, commands it says it ranNavigation onlyNot independent; never sufficient
L1 · StaticFormat, lint, typecheck, compile, schemaFast deterministic rejectionDoes not prove behavior
L2 · BehavioralUnit, integration, contract, UI flow, golden vectorsRequirements expressed as executable behaviorOnly covers modeled cases
L3 · NonfunctionalSecurity, accessibility, load, timing, power, thermalEngineering constraintsOften noisy or environment-dependent
L4 · Independent reviewFresh-context agent reviews requirement, diff, and evidenceMissing cases, scope, maintainability, threat reasoningModel-as-judge errors and correlated bias
L5 · Human/real-world gateOwner judgment, lab signoff, canary, user/production evidenceIrreversible, subjective, legal, safety, and material riskSlower and scarce

Protect the verifier from the maker

  • Mount acceptance tests and policy files read-only where practical.
  • Compare test/spec diffs and reject deleted, skipped, or weakened checks.
  • Give the reviewer immutable requirements, not only the maker summary.
  • Use a fresh context or separate agent for semantic review.
  • Run verification from a clean checkout at the candidate commit.
  • Capture exit code, stdout/stderr, version, duration, and artifact hash.
  • Mark flaky, skipped, unavailable, and inconclusive as distinct outcomes.
  • Make “verifier unavailable” fail closed for required gates.

Safe adoption path

LEVEL 0
Manual replay. A human runs the maker prompt, checks evidence, and records the decision. Optimize clarity before autonomy.
LEVEL 1
Agent makes; human verifies. The agent may edit an isolated branch, but humans choose retry/pass.
LEVEL 2
Automated verification. Controller runs protected deterministic checks; only passing candidates reach review.
LEVEL 3
Bounded retry. Permit 2–4 attempts with changed hypotheses, fixed permissions, and full audit state.
LEVEL 4
Event/scheduled discovery. Add triggers after stop behavior is reliable; open draft changes, never silently deploy.
LEVEL 5
Portfolio operation. Track success, escapes, cost, rollback, scope drift, and disable poorly performing loop classes.

Worked case study

CI repair: what three disciplined attempts look like

Signal: Windows integration test cancelDuringUpload times out after a retry refactor. Goal: restore cancellation within 2 seconds without weakening the test, changing public API, or adding an unbounded retry. Budget: 3 attempts, 45 minutes, 8 files, draft PR only.

DISCOVER
Controller deduplicates the failed check by task + commit + failure signature, creates a worktree, records baseline SHA and raw job log, then marks the run READY.
ATTEMPT 1
Maker reproduces locally and hypothesizes a missing cancellation check before retry sleep. Patch passes the target once but fails 2 of 20 stress runs. Verifier returns RETRYABLE with the two failing seeds.
ATTEMPT 2
New hypothesis: cancellation is checked, but the sleep itself is not interruptible. Maker replaces sleep with a cancellation-aware wait. All behavioral checks pass; independent review rejects a broad catch block that hides unrelated failures.
ATTEMPT 3
Maker narrows exception handling and adds the failing seed as a regression case. Target passes 20/20, affected suite and static checks pass, reviewer returns PASS.
HANDOFF
Controller opens a draft PR containing diff, hypotheses, command versions, raw results, known limitations, elapsed budget, and rollback note. A human still owns merge.
Worked loop attempts, new information, changes, evidence, and decisions
AttemptNew informationCandidate changeVerifier evidenceDecision
1Cancellation correlates with retry waitPre-wait cancellation check18/20 stress passesRetry: still flaky
2Wait must be interruptibleCancellation-aware delayBehavior passes; broad catch foundRetry: semantic risk
3Exception scope is the remaining defectNarrow catch + regression seedAll required layers passDraft PR

Why this converged: every retry came from new evidence, changed one hypothesis, preserved protected tests, and had an independent decision. Re-running attempt 1 with a more forceful prompt would not have been engineering.

Practical patterns

Loops that produce useful engineering evidence

CI repair loop

Trigger: failed check.
Maker: reproduce and patch.
Verify: regression + affected suite.
Stop: draft PR or repeated failure.

Dependency update loop

Trigger: scheduled scan.
Maker: one bounded update.
Verify: build, tests, API diff, security.
Stop: PR; major/breaking change escalates.

UI parity loop

Trigger: approved design.
Maker: implement one screen state.
Verify: screenshots, DOM, accessibility, flows.
Stop: thresholds met and designer review.

Deploy verification loop

Trigger: preview/pilot deployment.
Observe: smoke flows, errors, latency, health.
Decide: go, rollback, or human incident gate.

Firmware fault loop

Trigger: reproducible timeout trace.
Maker: one recovery hypothesis.
Verify: HIL fault injection, timing, power, recovery.
Stop: repeated target evidence.

FPGA optimization loop

Goal: throughput under correctness/PPA limits.
Maker: one architecture variant.
Verify: golden vectors, assertions, timing, resources, board benchmark.

Hardware rule: keep simulation, implementation reports, and physical measurements as separate verifiers. A loop must not treat simulation success as electrical, timing, thermal, or field signoff.
Threat model

The loop amplifies both capability and mistakes

Repeated autonomy increases exposure: the agent sees more untrusted inputs, calls more tools, and gets more chances to affect state. Model every external string—issue text, logs, web pages, test fixtures, dependency metadata, and review comments—as untrusted data, not instructions.

Loop Engineering threats, failure paths, controls, and mandatory stops
ThreatFailure pathControlMandatory stop
Prompt injectionIssue/log/web text tells the agent to reveal secrets or ignore policySeparate instructions from data; source labels; tool allowlists; no secret echoInstruction conflicts with loop policy
Verifier sabotageMaker skips, deletes, rewrites, or narrows testsProtected paths; baseline diff; read-only checks; clean verifier checkoutRequired verifier changed without approval
Secret exfiltrationLogs, patches, artifacts, or network calls expose credentialsShort-lived scoped secrets; redaction; egress allowlist; artifact scanningUnexpected secret access or outbound destination
Supply-chain compromiseLoop installs arbitrary package/action/tool to make progressLockfiles, pinned actions/versions, approved registries, dependency reviewNew source or unsigned/unreviewed executable required
Privilege creepAgent asks for more filesystem, connector, branch, or production accessLeast privilege; separate credentials; fail closed; human approvalAny capability expansion
State/branch raceConcurrent runs overwrite state or work on the same branchIdempotency key, lock/lease, atomic append log, one worktree per runOwnership/lease cannot be proven
Artifact spoofingOld or unrelated result is presented as current evidenceCommit SHA, command/version, timestamp, run ID, hashes, provenanceEvidence cannot be tied to candidate commit

Permission rule: retry logic must never increase authority automatically. A loop that cannot progress inside its original capability envelope is BLOCKED, not entitled to broader access.

Never run unattended when…

  • The goal is subjective, contradictory, or not testable.
  • The bug cannot be reproduced and evidence is mostly speculation.
  • Production, safety equipment, customer data, or regulated decisions are in scope.
  • Success requires legal, ethical, architecture, or product judgment.
  • The environment is unstable enough that pass/fail cannot be trusted.
  • The agent can modify its own policy, verifier, controller, or credential scope.
  • Rollback has not been proven or the action is irreversible.
  • A one-off supervised task is cheaper and clearer than automation.
Test and rollout

Evaluate the loop as a product, not a demo

A successful showcase proves possibility. A production decision needs a frozen task set, a baseline, repeated runs, failure injection, and operational metrics. Preserve transcripts and artifacts, then read samples—aggregate scores alone hide reward hacking and brittle behavior.

1 · DATASET
Create 20–50 representative tasks across easy, normal, ambiguous, unsafe, flaky, and impossible cases. Freeze inputs and acceptance criteria before tuning.
2 · BASELINE
Measure the current human workflow and a single supervised agent pass. Compare quality, time, intervention, and cost.
3 · ADVERSARIAL
Inject prompt-injection text, missing secrets, broken setup, flaky tests, dependency conflicts, stale state, and a required destructive action. The correct result is often safe refusal/escalation.
4 · SHADOW
Run against real signals without writing branches or external systems. Compare proposed decisions with what engineers actually did.
5 · CANARY
Enable one low-risk repository and one loop class with draft-only output, low concurrency, and immediate disable control.
6 · SCALE
Expand only if quality, escape rate, rollback, cost, and human attention improve. Retire loop classes that do not earn their operational burden.

Minimum scorecard

Accepted-task rateAccepted outputs ÷ attempted tasks, segmented by task class.
Verifier escape rateDefects found after the loop declared pass ÷ passed runs.
Convergence attemptsMedian and p95 maker attempts per accepted result.
Human interventionMinutes and number of handoffs per accepted result.
Cost per accepted changeModel + tools + CI + reviewer cost, including failed runs.
Scope amplificationChanged files/lines versus the smallest reviewed solution.
Rollback / reopen rateAccepted changes later reverted, hotfixed, or reopened.
Safe-stop accuracyUnsafe/impossible cases correctly blocked or escalated.
Evidence completenessRuns with required provenance, commands, results, and risks.
Flake sensitivityDecision variance across repeated identical cases.
promotion gate example:
accepted_task_rate ≥ baseline + target_gain
AND verifier_escape_rate ≤ baseline
AND safe_stop_accuracy = 100% on critical cases
AND p95 cost/time within budget
AND no automatic high-risk side effects

Readiness gate

Do not automate until these answers are clear

  • The goal is specific and testable.
  • Success and terminal failure states are explicit.
  • State survives restarts and context compaction.
  • Every run has a unique ID and isolated workspace.
  • The maker cannot silently alter protected verifiers.
  • At least one verifier observes real system behavior.
  • Flaky or inconclusive evidence is handled explicitly.
  • Attempts, time, cost, scope, and concurrency are capped.
  • Repeated failures escalate instead of looping forever.
  • Secrets, network, branches, and connectors use least privilege.
  • Irreversible changes require a human gate.
  • Every run leaves an auditable evidence package.
  • Rollback and cleanup are tested.
  • The loop can be paused or disabled immediately.
